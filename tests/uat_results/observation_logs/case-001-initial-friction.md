# UAT 典型失败案例：Case-001 (首步安装与执行摩擦)

## 1. 案例摘要
在 2026-04-02 的第一轮 UAT 测试中，所有非项目作者背景的测试者在执行“第一步：跑一条任务”时均未能成功，暴露出显著的环境与跨平台兼容性缺陷。

## 2. 详细表现与卡点分析

### A. 现象一：Windows PowerShell 语法失效 (Tester 2)
- **输入**：`python -m entrypoints.cli run --task "Search docs..."`
- **错误**：`命令语法不正确`。
- **根因**：PowerShell 对 `-m` 调用后的路径参数及双引号转义逻辑与标准 CMD 或 Bash 存在细微差异。
- **影响**：用户在第一步即被“封杀”，无法感受到产品的核心价值。

### B. 现象二：IO 或推理挂死 (Tester 1, 3)
- **输入**：`python -m entrypoints.cli run --task "Search docs for runtime context"`
- **表现**：bash 120s 超时返回，后台无明显心跳反馈。
- **分析**：
  - **IO 层面**：Windows 下 `python -m` 加载大模块可能触发杀毒软件扫描或磁盘 IO 瓶颈。
  - **逻辑层面**：默认测试任务涉及的“搜索”逻辑过重，导致非作者环境下的初次运行体验极差。
- **影响**：给用户一种“程序已死”的死板感，缺乏中间进度的可见性。

## 3. 纠偏建议 (已反馈兵部/工部)
- **命令统一化**：必须在文档中明确给出各 Shell 下的“复制即运行”指令。
- **Mock 级验证**：首个测试任务必须是“秒级返回”的极简任务，先建立用户信心。
- **心跳可见性**：长时任务需在终端提供类似 `(Searching...)` 的进度提示。

## 4. 硬验收标准参考
本案例已作为 **M3 硬验收（Hard Acceptance）** 的核心指标之一。新版本指南必须通过“在干净 Windows PowerShell 环境下 30s 内跑通极简任务”这一硬性测试。
